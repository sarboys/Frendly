0. Как проводить аудит
Приоритеты
Используй три уровня критичности:
P0 — критично: может привести к падению приложения, утечке данных, потере денег, неправильным подпискам, потере сообщений, дублированию оплат, обходу авторизации, блокировке интерфейса, деградации backend под нагрузкой.
P1 — важно: влияет на скорость, стабильность, UX, поддержку, тестируемость, качество кода, масштабирование.
P2 — улучшение: косметика, читаемость, мелкие оптимизации, рефакторинг без срочного риска.

1. Глобальные критерии качества
Производительность и отклик
 У приложения есть понятные performance budgets: старт приложения, переходы между экранами, открытие ленты, открытие встречи, отправка сообщения, загрузка голосового, поиск, фильтры.
 Нет тяжелых операций на UI-потоке Flutter.
 Нет синхронной обработки больших JSON, файлов, аудио, изображений на главном isolate.
 Нет лишних rebuild’ов виджетов.
 Нет лишних запросов при каждом открытии экрана.
 Нет повторной загрузки одних и тех же данных без причины.
 Нет N+1 запросов на backend.
 Нет N+1 запросов на клиенте.
 Нет неограниченных списков без пагинации.
 Нет неограниченной загрузки изображений/аватаров/аудио.
 Все тяжелые операции backend вынесены в очереди или workers.
 Backend не блокирует Node.js event loop CPU-тяжелыми задачами. Node.js официально предупреждает, что блокировка event loop или worker pool мешает обслуживать других клиентов.
Надежность
 Любое создание сущности защищено от дублей.
 Любая оплата/подписка обрабатывается идемпотентно.
 Любая отправка сообщения имеет client message id.
 Любое повторное нажатие кнопки не создает дубликат.
 Любой retry не ломает бизнес-логику.
 Любой offline/online переход корректно восстанавливает состояние.
 Любой WebSocket reconnect не дублирует подписки и события.
 Любое удаление/редактирование проверяет права.
 Любая пагинация стабильна при добавлении новых элементов.
 Любая фоновая задача имеет retry, timeout и dead-letter поведение.
Безопасность
 Есть проверка по OWASP ASVS для backend.
 Есть проверка по OWASP MASVS для мобильных приложений.
 Все входные данные валидируются.
 Backend не доверяет данным клиента.
 Авторизация проверяется на backend для каждого действия.
 Секреты не лежат в приложении.
 Токены защищены.
 Нет утечек персональных данных в логах.
 Нет возможности смотреть чужие встречи, чаты, голосовые, подписки, рейтинги.
 Нет IDOR-уязвимостей: доступ по чужому eventId, chatId, messageId, userId, subscriptionId.
OWASP ASVS используется как стандарт для проверки web/backend security controls, а OWASP MASVS — как отраслевой стандарт для мобильной безопасности, включая хранение данных, криптографию, аутентификацию, сетевое взаимодействие, качество кода и устойчивость к tampering.

 Back stack корректный после login/logout.

Два backend instances, два пользователя в одном чате на разных instances — сообщения доходят обоим.
WebSocket reconnect 20 раз подряд — нет дублей сообщений, unread, rooms, listeners.
Payment webhook приходит дважды — premium начисляется один раз.
Payment webhook приходит не по порядку — итоговый subscription state правильный.
Пользователя исключили из встречи, пока у него открыт чат — он больше не получает и не отправляет сообщения.
Logout во время активных API-запросов — старые responses не возвращают приватные данные в UI.
Пуш по удаленной встрече — приложение не падает, показывает корректный fallback.
Скролл 10 000 сообщений — нет memory leak, нет jank.
Лента 10 000 встреч с фильтрами — нет full reload, нет дублей, нет скачков.
Истекший access token у 20 параллельных запросов — refresh выполняется один раз.
Пользователь с истекшей подпиской пытается использовать premium API напрямую — backend отклоняет.
33. Итоговая формула хорошего результата
Код можно считать хорошо проверенным, если:
UI не блокируется тяжелыми задачами.
Все списки lazy, paginated, cache-aware.
Все critical actions идемпотентны.
Backend не доверяет клиенту.
Доступ проверяется на каждый объект.
Чат и WebSocket работают при reconnect, multi-device и multi-instance.
Голосовые не блокируют UI/backend.
Подписки проверяются сервером и переживают webhooks out-of-order.
БД защищена constraints, indexes и transaction boundaries.
Нет дублей в сообщениях, встречах, рейтингах, оплатах.
Есть нагрузочные тесты на самые опасные сценарии.
Есть observability: p95/p99, event loop lag, queue lag, WebSocket count, payment failures.
Есть тесты на плохую сеть, быстрые клики, logout, token expiry, удаление/бан/потерю доступа.

46. Находки аудита
Проблема:
Публичный dev login дает полный обход авторизации.
Где найдено:
`backend/apps/api/src/controllers/auth.controller.ts:11`
`backend/apps/api/src/services/auth.service.ts:16`
Почему это риск:
Любой внешний клиент может получить access token и refresh token для произвольного `userId`. Это прямой auth bypass.
Как воспроизвести:
Вызвать `POST /auth/dev/login` с телом `{ "userId": "user-me" }`.
Ожидаемое поведение:
Dev login должен быть отключен вне локальной разработки или жестко защищен по env-флагу.
Фактическое поведение:
Route публичный, ограничений по окружению нет.
Влияние: P0
Предложение исправления:
Убрать endpoint из продовой сборки. Минимум, включать его только при явном `ENABLE_DEV_AUTH=true` и только в non-production.
Нужен ли тест:
Да, нужен интеграционный тест, что route недоступен при production config.

Проблема:
OTP авторизация фактически отключена, код статичный, подсказка кода возвращается клиенту.
Где найдено:
`backend/apps/api/src/services/auth.service.ts:32`
`backend/apps/api/src/services/auth.service.ts:41`
Почему это риск:
Любой клиент может запросить challenge, увидеть `localCodeHint`, ввести `1111` и войти без подтверждения владения номером.
Как воспроизвести:

Ожидаемое поведение:
Код должен генерироваться случайно, храниться коротко, не возвращаться клиенту, а уходить через реальный delivery channel.
Фактическое поведение:
Код жестко зашит как `1111`, подсказка возвращается в ответе.
Влияние: P0
Предложение исправления:
Убрать `localCodeHint`, генерировать одноразовый код, добавить rate limit и попытки, dev stub держать только под отдельным env-флагом.
Нужен ли тест:
Да, нужны тесты на отсутствие code hint в non-dev режиме и на блокировку по попыткам.

Проблема:
Refresh flow на клиенте не single-flight, из-за rotation возможен шторм refresh и ложный logout.
Где найдено:
`mobile/lib/app/core/network/api_client.dart:30`
`mobile/lib/app/core/providers/core_providers.dart:75`
`backend/apps/api/src/services/auth.service.ts:124`
Почему это риск:

Ожидаемое поведение:
Один refresh на все ожидающие запросы.
Фактическое поведение:
Каждый 401 сам инициирует refresh.
Влияние: P1
Предложение исправления:
Сделать single-flight promise для refresh, а все ожидающие запросы подвешивать к нему.
Нужен ли тест:
Да, нужен concurrency test на 10-20 одновременных 401.
