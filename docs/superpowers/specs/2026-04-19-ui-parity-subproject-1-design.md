# Flutter UI Parity Subproject 1 Design

## Контекст

Полная работа по проекту разбита на 4 подпроекта:

1. дизайн система, темная тема, полный UI паритет всех экранов
2. auth, onboarding, profile, settings, media upload
3. events, join flow, host flow, check in, live, after party, stories, share
4. chats, realtime, people, personal chat, notifications, safety, report, match, paywall stubs

Этот документ покрывает только подпроект 1.

## Цель

Собрать во Flutter полное визуальное совпадение с `front/` для всех продуктовых экранов, включая светлую тему, темную тему, общие токены, общие виджеты, shell, навигационный ритм.

## Источники правды

Главный источник правды по виду:

- `front/src/index.css`
- `front/src/components/bigbreak/BigBreakApp.tsx`
- `front/src/components/bigbreak/BottomNav.tsx`
- `front/src/components/bigbreak/PhoneFrame.tsx`
- `front/src/components/bigbreak/screens/*`
- `docs/design-system-big-break.md`
- `docs/flutter-ui-mapping-big-break.md`
- `docs/flutter-engineering-standards.md`

React код не считается источником архитектуры Flutter.
`front/` нужен как живой визуальный эталон.

## Выбранный подход

Выбран вариант 2.

Сначала общая дизайн система.
Потом базовые виджеты.
Потом экраны.

Этот порядок нужен, чтобы:

- не расползлись цвета, радиусы, тени, размеры
- темная тема легла одинаково на все экраны
- экраны собирались из общих кирпичей, а не из локальных стилевых копий
- финальная сверка с `front/` была предсказуемой

## Что входит в подпроект 1

### 1. Основа темы

Во Flutter должны появиться две полные темы:

- `light`
- `dark`

Системный третий режим не нужен.
Светлая тема остается стартовой по умолчанию.
Темная включается только ручным тумблером в `Settings`.

Нужно перенести токены из `front/src/index.css` один в один:

- цвета поверхностей
- текстовые цвета
- primary и secondary палитру
- muted, border, input, ring
- chat bubble токены
- градиенты
- тени
- радиусы
- базовые размеры

Тема должна жить на уровне всего `MaterialApp`.
Переключение не должно ограничиваться одним экраном.

### 2. Общие виджеты

Перед переносом экранов нужно довести до эталона общие блоки:

- `bb_search_bar`
- `bb_chip`
- `bb_avatar`
- `bb_event_card`
- `bb_bottom_nav`
- `bb_chat_bubble`
- `bb_composer`
- `bb_pinned_meetup_card`

Если блок реально нужен минимум в двух местах, он живет в `shared/widgets`.
Если блок нужен одному сценарию, он остается внутри feature.

### 3. Shell и базовые поверхности

Нужно привести к эталону:

- viewport frame для desktop preview
- phone status bar
- root background
- tab shell
- push transition, которая уже используется в `go_router`

## Что не входит в подпроект 1

Не входит:

- полный backend для всех экранов
- перенос React логики как архитектуры Flutter
- автоконвертация JSX в Dart
- новый системный theme mode
- редизайн экранов во Flutter без обновления эталона в `front/`

## Архитектурные правила

### Source of truth

`front/` остается единственным источником правды по виду.
Если Flutter расходится с `front/`, правим Flutter, пока эталон сам не изменен.

### Theme ownership

Тема держится в одном месте.
Источник флага один, `darkMode`.
Экран `Settings` только меняет это состояние.

### Reuse boundaries

В `shared/widgets` выносятся только реальные общие блоки.
Нельзя тащить туда экранные секции ради формальной красоты.
Нельзя строить лишние базовые классы под тему.

### Screen assembly

После выравнивания токенов и общих блоков экраны собираются из них.
Локальные цвета, тени, радиусы, случайные размеры из экранов должны уйти.

### Performance baseline

Особое внимание нужно держать на:

- чатах
- длинных списках
- stories
- картах

Эти экраны должны не только выглядеть как эталон, но и оставаться плавными.

## Навигация

Навигация остается на `go_router`.

Табы только 4:

- `Tonight`
- `Chats`
- `People`
- `Profile`

Все остальные экраны открываются как full screen push.

Нижняя навигация:

- видна только на tab screens
- не участвует в layout push экранов

Auth flow, onboarding, layered screens из `front` не копируются как вебовая имитация слоя.
Во Flutter это обычные полные экраны, если для partial sheet нет отдельного эталона.

## Что считаем полным паритетом

Экран считается совпавшим только если выполнено все ниже:

- совпадает композиция экрана
- совпадают токены, цвет, шрифт, радиусы, тени, градиенты, высоты контролов, плотность
- совпадают состояния, default, pressed, active, unread, pending, typing, disabled, empty, loading
- совпадает навигационное поведение
- совпадает светлая тема
- совпадает темная тема
- экран не разваливается на узкой ширине, длинном тексте, длинных списках

Темная тема не считается отдельным редизайном.
Это тот же экран, тот же layout, тот же ритм, с токенами из `.dark` в `front/src/index.css`.

Работа идет в таком порядке:

1. токены, `AppTheme.light`, `AppTheme.dark`, viewport frame, status bar, shell surfaces

4. `Tonight`, `Chats`, `People`, `Profile`

## Проверка и приемка

Для каждого экрана приемка включает:

- ручную side by side сверку с `front/`
- сверку в light
- сверку в dark
- проверку базовых состояний

Desktop phone frame на macOS нужен для визуальной сверки.
Он не заменяет проверку safe area на реальном iOS и Android.

## Риски

### Движущийся эталон

`front/` может меняться во время переноса.
Если это случится, Flutter начнет догонять движущуюся цель.

### Неполностью проверенный dark в `front`

Темная тема задана токенами.
Часть экранов может быть еще не просмотрена вручную в dark.
Во Flutter нельзя додумывать эти места от себя.

### Локальные стилевые хвосты в `mobile/`

Во Flutter уже есть экраны с локальными стилями.
Если их не вычистить в пользу токенов, паритет расползется.

### Визуальный паритет без перформанса

Чаты, списки, карты, stories могут совпасть по виду, при этом просесть по плавности.
Для них одной скриншотной проверки мало.

## Ограничения

- источник правды по виду только `front/`
- автоперенос из React не используется
- подпроект 1 не закрывает весь backend
- единственный backend touch здесь, сохранение `darkMode`
- если для конкретного экрана нет отдельного dark решения, layout не меняется, меняются только токены темы
- новые shared блоки появляются только при реальном повторном использовании

## Критерии готовности подпроекта 1

Подпроект 1 считается завершенным только если:

- во Flutter есть полная `light` и `dark` тема, завязанная на общие токены
- `Settings` переключает тему всего приложения
- значение темы сохраняется и восстанавливается после перезапуска без мигания
- все продуктовые экраны из списка собраны с визуальным паритетом к `front/`
- общие виджеты приведены к единому виду
- в экранах не осталось случайных локальных цветов, теней, радиусов
- для ключевых токенов, темы, shell, общих виджетов есть тесты
- для каждого экрана пройдена ручная сверка в light и dark
- после завершения остается явный список отличий, если такие отличия вообще есть

## Результат подпроекта

После завершения подпроекта 1 команда получает:

- единый Flutter theme layer под `front`
- работающий dark mode на уровне всего приложения
- выровненные shared widgets
- полный визуальный паритет продуктовых экранов
- стабильную базу для следующих подпроектов, где уже будет доводиться backend логика
